# MedSim Frontend

Frontend-часть платформы `MedSim` для учебной симуляции процессов медицинской организации.

## Что уже реализовано

Сейчас фронт покрывает два связанных рабочих сценария: путь игрока от подключения к стартовой комнате до отдельного командного экрана после запуска игры, а также рабочее пространство ведущего для настройки сессии и последующего мониторинга команд.

### 1. Игрок подключается к существующей сессии

Игровой поток ориентирован в первую очередь на использование с телефона: крупные поля, вертикальная композиция и аккуратные карточки сессий на узких экранах.

Игрок:

- вводит имя;
- выбирает свою реальную должность в больнице;
- видит список доступных для подключения сессий;
- выбирает одну из созданных ведущим комнат;
- подключается к backend;
- после успешного входа попадает в стартовую комнату ожидания;
- после запуска сессии автоматически переключается на отдельный экран своей команды, не видя другие команды.

### 2. Ведущий входит и управляет игровой сессией

У ведущего уже есть отдельная панель, в которой можно:

- войти по логину и паролю;
- вручную создать игровую сессию по названию и количеству команд;
- получить автоматически сгенерированный код комнаты;
- получить автоматически созданные команды со смешными названиями в медицинской тематике;
- при необходимости переименовать уже созданную сессию, не меняя её код;
- переименовать команды внутри комнаты;
- вручную распределять игроков по командам;
- запускать автоматическое полное перераспределение игроков по командам с выравниванием состава;
- настраивать количество этапов;
- задавать время и режим каждого этапа;
- назначать роли вручную только участникам, у которых уже выбрана команда;
- запускать случайное распределение ролей отдельно внутри каждой команды;
- запускать, приостанавливать, продолжать, завершать и удалять сессию;
- видеть таймер и кнопки управления временем только после фактического старта игры;
- после старта переходить в отдельный мониторинг команд с краткой сводкой и детальным просмотром выбранной команды.

Суперпользователь пока остаётся UI-заглушкой: отдельный вход уже есть, но рабочий экран ещё не реализован.

## Технологии

- `React`
- `Vite`
- `TypeScript`
- CSS без UI-библиотеки
- `localStorage` для восстановления состояния интерфейса после перезагрузки

- адаптивная вёрстка с отдельной мобильной полировкой под телефоны

## Визуальный подход

Текущий интерфейс выдержан в минималистичном зелёном стиле, чтобы:

- визуально ассоциироваться с медицинской средой;
- не перегружать интерфейс;
- сохранить единый стиль между стартовой страницей, комнатой ожидания и панелью ведущего.

## Архитектурная идея

Структура фронта разбита по ролям ответственности:

- `components/` — визуальные части интерфейса;
- `constants/` — константы и справочные списки;
- `hooks/` — состояние и сценарная логика;
- `lib/` — локальная инфраструктура, например сохранение состояния приложения;
- `services/` — работа с backend API;
- `types/` — общие типы приложения, форм и ответов сервера.

Это позволяет спокойно развивать разные пользовательские потоки:

- игрок;
- ведущий;
- позже суперпользователь;
- затем игровые экраны по ролям.

## Backend API

По умолчанию фронт обращается к backend через относительный префикс `/api`. В режиме разработки `Vite` проксирует эти запросы на локальный backend `http://localhost:8080`, поэтому одной публичной ссылки на фронт достаточно и для работы с backend.

Если нужно обращаться к backend напрямую, это можно переопределить через `VITE_API_BASE_URL`.

Пример:

```env
VITE_API_BASE_URL=http://localhost:8080
```

Основные запросы сейчас сосредоточены в:

- `src/services/playerSessionsApi.ts`
- `src/services/gameSessionsApi.ts`
- `src/services/staffAuthApi.ts`

Для удалённого тестирования через `cloudpub` dev-сервер Vite уже настроен на:

- `server.host = 0.0.0.0`;
- `allowedHosts` для публичного домена;
- proxy `/api -> http://localhost:8080`.

## Текущий пользовательский поток

### Игрок

1. Открывает стартовую страницу.
2. Вводит имя и реальную должность.
3. Выбирает одну из доступных сессий, созданных ведущим.
4. После успешной отправки формы попадает в стартовую комнату.
5. Фронт периодически проверяет состояние сессии и назначение команды.
6. После запуска игры автоматически открывается отдельный экран команды игрока.
7. После перезагрузки страницы игрок остаётся в своей сессии, а интерфейс заново подтягивает актуальное состояние комнаты или команды.

На экране ожидания сейчас отображаются:

- код сессии;
- имя участника;
- реальная должность;
- игровая роль, если она уже назначена;
- команда, если она уже назначена;
- количество настроенных этапов;
- статус сессии.

После старта игрок видит отдельный командный экран, где доступны:

- только участники его собственной команды;
- его текущая игровая роль;
- название команды;
- конфигурация этапов, подготовленная ведущим.

### Ведущий

1. Выбирает служебный вход и профиль `Ведущий`.
2. Вводит логин и пароль.
3. Фронт проверяет учётные данные через backend endpoint `GET /api/auth/me`.
4. После успешной проверки открывается панель управления сессиями.
5. В панели можно:
   - создать сессию по названию и количеству команд;
   - увидеть автоматически созданные коды и команды;
   - открыть список сессий;
   - переименовать выбранную сессию;
   - открыть нужную сессию;
   - переименовать команды;
   - вручную распределить игроков по командам;
   - автоматически перераспределить всех игроков по командам заново и выровнять состав максимально поровну;
   - задать число этапов;
   - настроить длительность и режим каждого этапа;
   - распределить роли вручную только после выбора команды;
   - выполнить автоназначение ролей только для участников с командой и отдельно внутри каждой команды;
   - запустить сессию только после полной подготовки состава и этапов;
   - завершить сессию;
   - удалить сессию с подтверждением.

Во время подготовки в `LOBBY` таймер не показывается. Блок таймера и кнопки управления временем становятся видимыми только после старта игры. После запуска ведущий может поставить игру на паузу, продолжить её и завершить с подтверждением действия.
6. Пока сессия в `LOBBY`, ведущий работает в экране настройки команд, этапов и ролей.
7. После запуска сессии интерфейс переключается в отдельный мониторинг команд.
8. В мониторинге ведущий видит сводку по всем командам и может открыть детальный состав выбранной команды.
9. После перезагрузки страницы ведущий возвращается в свою панель и в последнюю открытую сессию, пока `Basic Auth` остаётся валидным.

Все запросы панели ведущего выполняются с `Basic Auth`.

### Реальные должности и игровые роли

Сейчас во фронте используются роли:

- `Главный врач`
- `Главная медсестра`
- `Главный инженер`
- `Сестра поликлинического отделения`
- `Сестра диагностического отделения`
- `Заместитель главного инженера по медтехнике`
- `Заместитель главного инженера по АХЧ`

Этот же список используется:

- на стартовой странице как выбор реальной должности участника;
- на панели ведущего как основа для назначения игровых ролей.

Отдельно остаётся вариант `Другая должность`, если участник не попадает в этот список.

### Правила команд и ролей

На фронте отражены ограничения backend-логики:

- команды создаются автоматически при создании сессии;
- ведущий может переименовывать команды уже внутри комнаты;
- авторазбиение по командам при каждом нажатии заново пересобирает весь состав команд и устраняет перекосы;
- роли можно выдавать только тем участникам, у которых уже выбрана команда;
- автоназначение ролей выполняется отдельно внутри каждой команды;
- в каждой команде должно быть минимум `3` участника для автоделения ролей;
- в каждой команде гарантируются роли:
  - `Главный врач`
  - `Главная медсестра`
  - `Главный инженер`
- распределение не назначает игроку роль, совпадающую с его реальной должностью;
- внутри одной команды игровые роли не повторяются;
- остальные участники команды получают одну из исполнительских ролей:
  - `Сестра поликлинического отделения`
  - `Сестра диагностического отделения`
  - `Заместитель главного инженера по медтехнике`
  - `Заместитель главного инженера по АХЧ`
- если в конкретной команде меньше `3` участников или больше `7`, backend вернёт ошибку и не выполнит автоделение ролей;
- старт игры невозможен, пока все игроки не распределены по командам;
- старт игры невозможен, пока в каждой команде не назначены `Главный врач`, `Главная медсестра` и `Главный инженер`.

## Мобильный сценарий

Сейчас интерфейс отдельно подправлен под телефоны:

- уменьшены отступы и заголовки на узких экранах;
- карточки, формы и кнопки перестраиваются в одну колонку;
- длинные названия ролей, должностей и команд переносятся без горизонтального скролла;
- увеличены tap-target для кнопок и полей ввода;
- список доступных сессий и форма подключения удобно читаются на мобильной ширине;
- панель ведущего и мониторинг команд остаются рабочими на мобильной ширине, хотя основной сценарий для ведущего всё же удобнее на ноутбуке.
